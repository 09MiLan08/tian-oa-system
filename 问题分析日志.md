# 问题分析日志
## 一、《蓉城游戏开发者技术社区（Vue3多端响应博客）》
### 问题1：路由配置错误
- 出现场景：前端开发（刚学配置页面跳转的时候）
- 问题现象：点页面上的链接跳不动，还提示“路由不存在”，控制台一堆红色报错
- 排查过程：打开路由配置文件，对着教程一行行看,发现自己写的跳转路径和配置里的不一样,才想起新增的页面没加到路由列表里
- 根因分析：1. 把“/article”写成“/artcle”，少个字母都没发现2. 以为新建页面就能直接跳，不知道得手动在路由里加上3. 路由名随便起的，结果和跳转时用的对不上
- 解决方案：1. 把写错的路径改过来2. 新增页面后去路由文件里加一行，加完刷新页面试试能不能跳
- 优化启示： 配置完路由点一下链接测试2. 使用标准化命名
- 日期：2025-10-24

### 问题2：多端响应式布局错乱
- 出现场景：前端开发
- 问题现象：电脑上看着好好的页面，用手机打开就乱了，按钮跑屏幕外面去了，文字也挤在一起
- 排查过程：1. 检查样式里的尺寸，发现全写的“px”2. 问了大哥才知道手机端不能用固定尺寸
- 根因分析：1. 以为用px写就行，没想到手机屏幕大小不一样 2. 不知道还有vw、vh这种能跟着屏幕变的单位3. 误以为“响应式”就是自动适配，不用额外设置
- 解决方案：1. 把所有“px”改成“vw”，比如100px改成5vw，屏幕想大就大，屏幕想小就小 2. 在页面头部加了一行代码：`<meta name="viewport" content="width=device-width, initial-scale=1.0">`，手机就不会乱缩放了
- 优化启示：1. 一开始就想好要做手机端，别等电脑端做完了再改，返工太麻烦了2. 开发时一边做一边用手机模拟器看效果3. 复杂的布局用现成的工具就是比自己写省事多了
- 日期：2025-10-28

### 问题3：组件通信props传递失败
- 出现场景：前端开发（做父子组件传数据的时候）
- 问题现象：父组件想给子组件传个文章ID，结果子组件拿不到，打印出来是“undefined”，完全懵了
- 排查过程：1. 检查父组件传数据的名字，是“ArticleId”再 看子组件接收的名字，写成了“articleid” 2. 发现数据是从接口拿的，可能还没拿到就传过去了
- 根因分析：1. 大小写没分清，以为“ArticleId”和“articleid”是一回事2. 子组件没说清楚要接收什么类型的数据，传过去的是数字，它按字符串收3. 误以为接口数据一调用就有，没等数据加载完就渲染子组件了
- 解决方案：1. 子组件里明确写清楚要接收的类型，2. 加判断，数据加载完了再显示子组件，没加载完就显示“加载中”3. 复杂的数据用叫Pinia的工具
- 优化启示：1. 接口数据要等它加载完再用，别着急渲染2. 组件多了之后，用专门的工具管理数据，比一层层传更方便
- 日期：2025-11-05

### 问题4：打包后页面加载缓慢
- 出现场景：部署上线
- 问题现象：传到服务器上打开页面要等好几秒，图片还加载半天
- 排查过程：1. 找了个工具看哪些文件占空间，发现ECharts占了一半
- 根因分析：1. 不知道打包的时候能去掉没用的代码，导致文件冗余2.打 开了个叫sourceMap的东西，生成了很多额外文件
- 解决方案：1. 关掉sourceMap，文件瞬间小了很多2. 让服务器打开压缩使传输的时候文件更小
- 优化启示：1. 打包前先看看哪些文件占空间，针对性优化2. 不乱开配置项，没用的功能关掉能省不少空间
- 日期：2025-11-05

## 二、《成都城市生活数据洞察平台（FastAPI和ECharts）》
### 问题1：接口参数校验失败
- 出现场景：后端开发（写接口给前端调用的时候）
- 问题现象：前端调用接口就报错“422”，参数不对
- 排查过程：1. 接口要数字，前端传的参数传了字符串2. 发现接口要求传“region”参数，前端忘了传 3. 日期格式前端传的是“2025-11-08”，接口是另一种格式
- 根因分析：1. 前端没看清楚参数要什么类型2. 接口没设默认值，少传一个参数就报错了3. 以为前端会自己校验参数，没在接口里加详细说明
- 解决方案：1. 看清前端和接口文档，把参数类型、格式都对齐2. 给必传参数加了默认值，比如没传“region”就默认“成都”3. 用工具明确写清楚参数要求，比如“必须是数字，范围1-100”3. 调用前端前先检查参数
- 优化启示：1. 复杂参数要写清楚规则，方便前端对照2. 前端提前校验参数，能少很多无效请求3. 接口文档要实时更新
- 日期：2025-11-08

### 问题2：CSV文件读取编码错误
- 出现场景：后端开发（读取数据文件的时候）
- 问题现象：用pandas读CSV文件，直接报错“UnicodeDecodeError”，说解码失败，完全不知道啥意思
- 排查过程：1. 用记事本打开CSV文件，发现是Windows保存的2. 文件编码分UTF-8和GBK
- 根因分析：1. 读文件的时候没指定编码，默认用UTF-8读GBK文件直接错了
- 解决方案：1. 读文件的时候加上编码参数 2. 把所有CSV文件都转成UTF-8格式
- 优化启示：1. 清楚编码格式；2. 读文件一定要指定编码，不依赖默认值
- 日期：2025-11-12

### 问题3：ECharts图表无法渲染
- 出现场景：前端开发（做数据可视化图表的时候）
- 问题现象：接口数据拿到了，但图表就是不显示，控制台说“data is not an array”，以为是接口返回错了
- 排查过程：1. 打印接口返回的数据，发现是字典格式，而图表要数组2. 图表需要“value”字段，接口返回的是“count”3. 没数据的时候图表直接初始化，导致报错
- 根因分析：1. 没确认后端数据格式，以为后端会以图表要求返回2. 不知道图表对数据格式有要求，以为随便传数据就行3. 没考虑到没数据的情况，图表没数据就渲染失败4. 以为接口返回的数据能直接用，不用处理
- 解决方案：1. 让后端把数据改成数组格式，方便图表使用2. 自己配个小函数，把“count”改成“value”，适配图表3. 没数据的时候显示“暂无数据”，不让图表空着初始化4. 把图表渲染的代码封装，以后直接用，不用重复写
- 优化启示：1. 图表不显示时先看数据格式对不对，再看代码有没有错
- 日期：2025-11-12

### 问题4：静态文件访问404
- 出现场景：后端开发（配置静态文件的时候）
- 问题现象：想看静态文件夹里的HTML页面，结果报“404”，找不到文件，明明文件就在文件夹里
- 排查过程：1. 检查代码，发现没配置静态文件路径2. 配置完发现访问路径不对，配置的是“/static”，访问用了“/assets”3. 文件名有大写字母，服务器上区分大小写，导致找不到
- 根因分析：1. 以为静态文件放文件夹里就能直接访问，不知道要配置2. 配置路径和访问路径没统一3. 文件名随便起，大小写不一，没考虑服务器情况4. 配置完没测试，直接访问前端，出了问题才排查
- 解决方案：1. 在代码里加了静态文件配置，把“static”文件夹挂载上去2. 访问路径和配置路径保持一致，都用“/static”3. 把所有文件名改成小写加横杠
- 优化启示：1. 项目一开始就配置好静态文件2. 文件名和路径要统一规范3. 配置任何东西后，先测试一遍3. 服务器上文件名区分大小写
- 日期：2025-11-20

## 三、《天府软件园企业OA管理平台（Django和MySQL）》
### 问题1：数据库连接失败
- 出现场景：后端开发（项目刚初始化的时候）
- 问题现象：启动项目就报错，说“Access denied for user 'root'@'localhost'”，数据库连不上，完全不知道咋回事
- 排查过程：1. 检查数据库密码，发现配置文件里的密码和实际密码不一样2. 发现MySQL服务没启动
- 根因分析：1. 数据库密码记混了，配置文件里写的是旧密码
- 解决方案：1. 把配置文件里的密码改成正确的
- 优化启示：1. 要记好数据库密码2. 配置完数据库一定要测试连接，避免白忙活
- 日期：2025-11-25

### 问题2：数据库迁移冲突
- 出现场景：后端开发（修改数据模型之后）
- 问题现象：执行迁移命令就报错，说“Duplicate entry 'xxx' for key 'PRIMARY'”，主键重复，迁移不进去
- 排查过程：1. 检查数据模型，发现没指定主键2. 查看数据库表，里面已经有数据3. 发现之前手动改过表结构，和迁移文件对不上
- 根因分析：1. 手动改了数据库表结构，导致迁移文件和实际表对不上2. 重复执行迁移命令，导致数据重复
- 解决方案：1. 在数据模型里明确加了自增主键，比如“id = models.AutoField(primary_key=True)”2. 先备份数据库里的数据，然后删掉冲突的重复数据3. 把迁移文件全删了（除了__init__.py），重新生成新的迁移文件4. 重新执行迁移命令，这次成功了
- 优化启示：1. 建数据模型的时候就自己指定主键，不依赖Django自动生成2. 别手动改数据库表结构，要改就通过修改模型再生成迁移文件3. 迁移失败时先看是不是数据冲突，再看迁移文件有没有问题
- 日期：2025-11-25

### 问题3：权限控制失效
- 出现场景：后端开发（做登录权限控制的时候）
- 问题现象：没登录的人也能直接访问需要登录的页面，审批功能没起到限制作用
- 排查过程：1. 检查视图函数，发现没加登录验证的代码2. 看配置文件，发现少了个认证相关的中间件3. 测试登录后的数据，发现跨域的时候登录状态没保存
- 根因分析：1. 以为只要做了登录页面，就自动有权限控制，不知道要给视图加单独权限2. 配置文件里漏装了认证中间件，Django没法识别登录状态3. 前后端分离项目用了session保存登录状态，跨域的时候session失效
- 解决方案：1. 在需要登录的视图函数上加装饰器，比如“@login_required”，没登录就能跳转到登录页2. 在配置文件里补全了认证中间件3. 加JWT的方式保存登录状态，生成一个token，前端每次请求都带着
- 优化启示：1. 权限控制不是自动有的，要给每个需要的视图加单独权限2. 前后端分离项目用token更靠谱；3. 开发完一定要测试未登录能不能访问4. 重要功能（比如审批、数据修改）一定要加权限，避免普通用户随便操作
- 日期：2025-11-30
